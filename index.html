Module 01

Teradata Basics

Understand what Teradata is, why it exists, how it differs from OLTP databases, and its foundational concepts that everything else builds upon.

● Beginner 4 Topics
Introduction

What is Teradata?

Teradata is a Relational Database Management System (RDBMS) designed specifically for large-scale analytical workloads — OLAP (Online Analytical Processing), not OLTP.

It uses Massively Parallel Processing (MPP) to distribute data and queries across hundreds/thousands of independent processing units.

Key Differentiators

  • Shared-Nothing MPP architecture
  • Linear scalability (add nodes = more power)
  • Automatic data distribution via hashing
  • Full SQL-92 compliance + extensions
  • Built-in parallel optimizer
  • Handles petabytes natively

Teradata Architecture — Big Picture

🧠 Parsing Engine (PE)

  • Receives SQL from client
  • Parses & validates SQL syntax
  • Calls the Optimizer to find best execution plan
  • Dispatches work steps to AMPs via BYNET
  • Collects results and returns to client

⚡ Access Module Processors (AMPs)

  • Core processing units — each manages its own disk (vdisk)
  • Each AMP stores a subset of table rows
  • AMPs work in parallel — no shared memory
  • Handles read/write, sorting, aggregation
  • More AMPs = more parallelism = faster queries

🔗 BYNET

  • Internal high-speed interconnect network
  • Connects PEs to AMPs
  • Redundant — BYNET 0 and BYNET 1
  • Handles redistribution of rows during joins

💻 Node

  • Physical/virtual server in the cluster
  • Each node has multiple AMPs and a PE
  • Nodes are independent (shared-nothing)
  • Adding nodes scales both compute and storage
Data Flow: How a Query Executes
1
Client submits SQL

Application / BTEQ sends SQL to the Parsing Engine via JDBC/ODBC/CLI

2
PE parses & optimizes

Parser validates syntax. Optimizer uses statistics to build the best execution plan (join order, strategy, etc.)

3
PE dispatches steps to AMPs via BYNET

Execution plan broken into steps. Each AMP receives its work package via the BYNET interconnect.

4
AMPs work in parallel

Every AMP processes its local rows simultaneously. Aggregations, sorts, joins all happen in parallel.

5
Results returned to PE → Client

AMPs send partial results back to PE via BYNET. PE merges final result and returns to client.

💡
Key Insight: The power of Teradata is that steps 4 happens on ALL AMPs simultaneously. A 512-AMP system processes data ~512x faster than a single-processor system for full table scans. This is why large aggregations are extremely fast.
Fundamental Teradata Concepts

Row Hashing

When a row is inserted, Teradata applies a hash function to the Primary Index value. The resulting hash determines which AMP stores the row. Same PI value → always same AMP.

⚡
This is why choosing the right PI is so critical — a bad PI = uneven distribution = skew = slow queries.

Primary Index (PI)

  • UPI — Unique Primary Index (no duplicates, even distribution)
  • NUPI — Non-Unique PI (duplicates allowed, may cause skew)
  • PPI — Partitioned PI (range-based access within AMP)
  • PI is NOT necessarily the same as primary key
  • Choose: high cardinality, frequently joined, evenly distributed column(s)

Secondary Indexes

  • USI — Unique Secondary Index — subtable stored on every AMP
  • NUSI — Non-Unique Secondary Index — bitmap-based
  • Secondary indexes add storage overhead
  • Useful for frequently filtered non-PI columns

Space Types

  • PERM Space — Permanent storage for tables/indexes
  • SPOOL Space — Temp work area for query results & intermediate steps
  • TEMP Space — For Global Temporary Tables
  • PERM and SPOOL are per-user/per-database allocations

Data Definition Language
Teradata SQLCREATE TABLE — Full Syntax
-- Standard Permanent Table
CREATE SET TABLE EDW_PROD.SALES_FACT
(
    SALE_ID      INTEGER         NOT NULL,
    CUST_ID      INTEGER         NOT NULL,
    PROD_ID      INTEGER         NOT NULL,
    SALE_DATE    DATE FORMAT 'YYYY-MM-DD' NOT NULL,
    AMOUNT       DECIMAL(15,2),
    REGION_CD    CHAR(3),
    LOAD_DT      DATE            DEFAULT DATE
)
PRIMARY INDEX (CUST_ID)           -- NUPI: rows distributed by CUST_ID hash
PARTITION BY RANGE_N(             -- PPI: partition by year-month
    SALE_DATE BETWEEN
    DATE '2020-01-01' AND
    DATE '2030-12-31'
    EACH INTERVAL '1' MONTH
);

-- Volatile Table (session-scoped temp)
CREATE VOLATILE TABLE VT_TEMP_SALES
(
    SALE_ID  INTEGER,
    AMOUNT   DECIMAL(12,2)
) PRIMARY INDEX (SALE_ID)
ON COMMIT PRESERVE ROWS;

-- SET vs MULTISET: SET rejects duplicate rows, MULTISET allows
-- Always use SET for fact tables to prevent accidental duplicates

Table Types in Teradata

  • Permanent Table — Persists until dropped
  • Volatile Table (VT) — Exists only for session. Auto-dropped on disconnect.
  • Global Temporary Table (GTT) — Structure persists; data is session-scoped
  • Derived Table — Inline sub-query result set
  • View — Named SELECT statement

SET vs MULTISET

SET table: Rejects exact duplicate rows (all column values identical). Teradata default.

MULTISET table: Allows duplicate rows. Use when source data intentionally has duplicates.

Use SET tables in EDW to enforce data integrity and prevent accidental load duplicates.

DML — Data Manipulation Language
Teradata SQLINSERT / UPDATE / DELETE / MERGE
-- INSERT from SELECT (ETL pattern)
INSERT INTO EDW_PROD.SALES_FACT
SELECT
    s.SALE_ID,
    s.CUST_ID,
    s.PROD_ID,
    CAST(s.SALE_DT AS DATE FORMAT 'YYYY-MM-DD'),
    s.SALE_AMT,
    s.REGION,
    DATE
FROM EDW_STG.SALES_STG s
WHERE s.LOAD_DT = DATE - 1
  AND s.STATUS_CD = 'A';

-- UPDATE with JOIN
UPDATE f
FROM EDW_PROD.SALES_FACT f,
     EDW_PROD.CUST_DIM c
SET f.REGION_CD = c.REGION_CD
WHERE f.CUST_ID = c.CUST_ID
  AND f.REGION_CD IS NULL;

-- DELETE with condition
DELETE FROM EDW_PROD.SALES_FACT
WHERE SALE_DATE < DATE - 365
  AND REGION_CD = 'XX';

-- MERGE (Upsert) — Most used in SCD Type 1
MERGE INTO EDW_PROD.CUST_DIM tgt
USING EDW_STG.CUST_STG src
ON (tgt.CUST_ID = src.CUST_ID)
WHEN MATCHED THEN UPDATE SET
    tgt.CUST_NAME = src.CUST_NAME,
    tgt.EMAIL     = src.EMAIL,
    tgt.UPD_DT    = DATE
WHEN NOT MATCHED THEN INSERT
VALUES (src.CUST_ID, src.CUST_NAME, src.EMAIL, DATE);
SELECT — Advanced Patterns
Teradata SQLSELECT patterns used in EDW
-- Date arithmetic (Teradata-specific)
SELECT SALE_DATE,
       SALE_DATE - 30          AS THIRTY_DAYS_AGO,
       DATE                    AS TODAY,
       EXTRACT(YEAR FROM SALE_DATE) AS SALE_YEAR
FROM EDW_PROD.SALES_FACT
WHERE SALE_DATE BETWEEN DATE - 30 AND DATE;

-- QUALIFY — filter on window function (Teradata-specific)
SELECT CUST_ID, SALE_DATE, AMOUNT
FROM EDW_PROD.SALES_FACT
QUALIFY ROW_NUMBER() OVER(PARTITION BY CUST_ID ORDER BY SALE_DATE DESC) = 1;

-- TOP with SAMPLE
SELECT TOP 100 * FROM EDW_PROD.SALES_FACT
ORDER BY AMOUNT DESC;

-- Conditional aggregation
SELECT
    REGION_CD,
    COUNT(*) AS TOTAL_SALES,
    SUM(CASE WHEN AMOUNT > 1000 THEN 1 ELSE 0 END) AS HIGH_VALUE,
    SUM(AMOUNT) AS TOTAL_AMOUNT,
    AVG(AMOUNT) AS AVG_AMOUNT
FROM EDW_PROD.SALES_FACT
WHERE SALE_DATE >= DATE - 90
GROUP BY 1
HAVING TOTAL_AMOUNT > 100000
ORDER BY TOTAL_AMOUNT DESC;

Teradata SQLJoin Types

SELECT f.SALE_ID, c.CUST_NAME, p.PROD_NAME, f.AMOUNT
FROM EDW_PROD.SALES_FACT f
INNER JOIN EDW_PROD.CUST_DIM  c ON f.CUST_ID = c.CUST_ID

INTERSECT

Analytics (Window) Functions
🎯
Most asked in interviews! Analytics functions compute values across a set of rows (the "window") without collapsing them like GROUP BY does. Used heavily in SCD Type 2, ranking, running totals.
Teradata SQLWindow Functions — Complete Reference
-- ROW_NUMBER: Unique sequential number per partition
SELECT CUST_ID, SALE_DATE, AMOUNT,
    ROW_NUMBER() OVER(PARTITION BY CUST_ID ORDER BY SALE_DATE DESC) AS RN
FROM EDW_PROD.SALES_FACT;

-- QUALIFY to get latest record per customer (SCD2 pattern)
SELECT CUST_ID, SALE_DATE, AMOUNT
FROM EDW_PROD.SALES_FACT
QUALIFY ROW_NUMBER() OVER(PARTITION BY CUST_ID ORDER BY SALE_DATE DESC) = 1;

-- RANK / DENSE_RANK
SELECT CUST_ID, AMOUNT,
    RANK()       OVER(ORDER BY AMOUNT DESC) AS RNK,      -- gaps: 1,2,2,4
    DENSE_RANK() OVER(ORDER BY AMOUNT DESC) AS DRNK     -- no gaps: 1,2,2,3
FROM EDW_PROD.SALES_FACT;

-- Running Total / Cumulative Sum
SELECT SALE_DATE, AMOUNT,
    SUM(AMOUNT) OVER(ORDER BY SALE_DATE
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS RUNNING_TOTAL
FROM EDW_PROD.SALES_FACT;

-- LAG / LEAD: Compare with previous/next row
SELECT CUST_ID, SALE_DATE, AMOUNT,
    LAG(AMOUNT,1,0) OVER(PARTITION BY CUST_ID ORDER BY SALE_DATE) AS PREV_AMOUNT,
    AMOUNT - LAG(AMOUNT,1,0) OVER(PARTITION BY CUST_ID ORDER BY SALE_DATE) AS DELTA
FROM EDW_PROD.SALES_FACT;

-- PERCENTILE / NTILE
SELECT CUST_ID, AMOUNT,
    NTILE(4) OVER(ORDER BY AMOUNT) AS QUARTILE
FROM EDW_PROD.SALES_FACT;
Macros & Stored Procedures

Macros

A macro is a saved SQL statement or series of statements with optional parameters. Executed with EXEC.

  • Compiled once, executed many times
  • Can contain multiple SQL statements
  • All statements run as a transaction
  • Supports input parameters
  • No conditional logic (use SP for that)

Stored Procedures

Procedural code that runs on Teradata server. Supports variables, loops, conditions, error handling.

  • Full procedural logic (IF/WHILE/FOR)
  • IN/OUT/INOUT parameters
  • Dynamic SQL execution
  • Error handling with CONDITION HANDLER
  • Used for complex ETL transformation logic
Teradata SQLMacro & Stored Procedure Examples
-- CREATE MACRO with parameter
REPLACE MACRO EDW_PROD.GET_CUST_SALES
(p_cust_id INTEGER)
AS (
    SELECT SALE_DATE, AMOUNT, REGION_CD
    FROM EDW_PROD.SALES_FACT
    WHERE CUST_ID = :p_cust_id
    ORDER BY SALE_DATE DESC;
);

-- Execute Macro
EXEC EDW_PROD.GET_CUST_SALES(12345);

-- ─────────────────────────────────────────────
-- CREATE STORED PROCEDURE
REPLACE PROCEDURE EDW_PROD.SP_LOAD_DAILY_SALES
(IN p_load_date DATE,
 OUT p_rows_loaded INTEGER)
BEGIN
    DECLARE v_count INTEGER DEFAULT 0;

    -- Delete existing data for the date
    DELETE FROM EDW_PROD.SALES_FACT
    WHERE SALE_DATE = :p_load_date;

    -- Insert from staging
    INSERT INTO EDW_PROD.SALES_FACT
    SELECT * FROM EDW_STG.SALES_STG
    WHERE LOAD_DATE = :p_load_date;

    SELECT COUNT(*) INTO :v_count
    FROM EDW_PROD.SALES_FACT
    WHERE SALE_DATE = :p_load_date;

    SET :p_rows_loaded = :v_count;
END;

-- Call Stored Procedure
CALL EDW_PROD.SP_LOAD_DAILY_SALES(DATE - 1, rows_out);
Module 03

Architecture Deep Dive

Master Primary Index design, data distribution, space management, and the Teradata optimizer. These topics dominate senior-level interviews.

● Intermediate 4 Topics
Primary Index — The Most Critical Design Decision
🎯
The Golden Rule: Every table in Teradata MUST have a Primary Index. The PI determines which AMP stores each row. Choose wrong → data skew → slow queries.

UPI — Unique Primary Index

  • No duplicate PI values allowed
  • Guarantees perfectly even distribution
  • 1 AMP access for PI-based lookups (fastest)
  • Best for dimension tables (CUST_ID, PROD_ID)
  • Use when column is already a natural unique key

NUPI — Non-Unique Primary Index

  • Duplicate PI values allowed
  • May cause skew if low cardinality
  • Used for fact tables joined to dimensions
  • Choose high cardinality column(s) for even distribution
  • Example: CUST_ID on SALES_FACT (one customer → many sales)

PPI — Partitioned Primary Index

  • Adds partition elimination on top of PI
  • Best for large fact tables queried by date range
  • Query touches only relevant partitions — dramatic speed improvement
  • Usually: partition by month/year + PI on join column
  • Max 65,535 partitions per table

Secondary Indexes

  • USI: Unique secondary index — creates subtable on every AMP. 2 AMP accesses.
  • NUSI: Non-unique secondary index — bitmap index within AMP. Full AMP scan with filter.
  • Add index for frequently filtered non-PI columns
  • Trade-off: faster reads, slower DML (index maintained)

How Teradata Distributes Data
1
PI value extracted from row

When a row is inserted, Teradata extracts the Primary Index column value(s)

2
Hash function applied

A 32-bit hash is computed on the PI value. Same PI value always produces same hash — deterministic.

3
Hash maps to AMP

The hash is divided into buckets (DSW - Data Segment Wheel). Each AMP owns a range of buckets. Row goes to the AMP owning that bucket.

4
Row stored within AMP

Within the AMP, rows are further sorted by row hash for efficient retrieval.

Skew — The #1 Performance Killer

What is Skew?

Skew occurs when data is unevenly distributed across AMPs. Some AMPs have significantly more rows than others.

The slowest AMP determines query completion time — other AMPs wait idle while the hot AMP finishes.

How to Detect Skew

  • Use EXPLAIN plan — check "skew" warnings
  • Check AMP distribution via system tables
  • Query: HELP STATS tablename;
  • Viewpoint workload monitor shows hot AMPs
Teradata SQLCheck skew via system tables
-- Check AMP-level row distribution
SELECT
    HASHAMP(HASHROW(CUST_ID)) AS AMP_NUM,
    COUNT(*) AS ROW_COUNT
FROM EDW_PROD.SALES_FACT
GROUP BY 1
ORDER BY 2 DESC;

-- Skew factor query
SELECT
    TABLENAME,
    MAX(CURRENTPERM) AS MAX_AMP_PERM,
    AVG(CURRENTPERM) AS AVG_AMP_PERM,
    (MAX(CURRENTPERM) / AVG(CURRENTPERM) - 1) * 100 AS SKEW_PCT
FROM DBC.TABLESIZE
WHERE DATABASENAME = 'EDW_PROD'
GROUP BY 1
HAVING SKEW_PCT > 10
ORDER BY SKEW_PCT DESC;
Parsing Engine & Optimizer

How the Optimizer Works

Teradata's cost-based optimizer (CBO) evaluates multiple execution plans and chooses the lowest-cost one based on:

  • Statistics — Column value distribution, row counts, unique values
  • Index information — PI, USI, NUSI availability
  • Join order — Small table first in nested joins
  • Join strategy — Row hash match, merge, hash, product
  • Spool estimates — Intermediate result sizes
⚠️
Without statistics, the optimizer guesses! Stale or missing statistics is the most common cause of bad execution plans. Always COLLECT STATS on join columns, filter columns, and PI columns.

Performance Tuning

Read EXPLAIN plans, collect statistics, detect and fix skew, and apply tuning strategies that separate senior engineers from juniors.

● Advanced 4 Topics
Reading EXPLAIN Plans
💡
Prepend any SELECT with EXPLAIN to see the execution plan without running the query. This is your most powerful tuning tool in Teradata.
Teradata SQLEXPLAIN output interpretation
-- To get explain plan, just prepend EXPLAIN:
EXPLAIN
SELECT c.CUST_NAME, SUM(f.AMOUNT) AS TOTAL
FROM EDW_PROD.SALES_FACT f
JOIN EDW_PROD.CUST_DIM c ON f.CUST_ID = c.CUST_ID
WHERE f.SALE_DATE >= DATE - 30
GROUP BY 1;

/* Sample EXPLAIN output — what to look for:

 1) "We do an all-AMPs RETRIEVE step..."
    → Full table scan. Expected for large fact tables.
    → If unexpected, check if stats are collected.

 2) "...with a residual condition of..."
    → Filter pushed down. Good.

 3) "...redistributing rows of [table] by hash code..."
    → BYNET traffic! One table PI doesn't match join key.
    → Consider changing PI or adding USI.

 4) "...with no residual conditions (confidence: high/low)..."
    → "low" = optimizer is guessing = run COLLECT STATS!

 5) "Product Join"
    → Cartesian product — usually unintentional! Fix immediately.

 6) "Estimated rows: 1 (no statistics available)"
    → Missing stats = optimizer blindly guessing row counts.
*/

    COLUMN (SALE_DATE),
    COLUMN (REGION_CD),
    COLUMN (CUST_ID, SALE_DATE)  -- multi-column stats
ON EDW_PROD.SALES_FACT;

-- Index statistics
COLLECT STATISTICS
    INDEX (CUST_ID)
ON EDW_PROD.SALES_FACT;

-- View existing statistics
HELP STATISTICS EDW_PROD.SALES_FACT;

-- Drop statistics
DROP STATISTICS
    COLUMN (REGION_CD)
ON EDW_PROD.SALES_FACT;

-- When to collect stats:
-- 1. After initial table creation and load
-- 2. After major data changes (>10% change)
-- 3. ALWAYS on PI columns, join columns, filter columns
-- 4. After adding/dropping partitions
Skew Detection & Resolution

Symptoms of Skew

  • One AMP takes much longer than others
  • EXPLAIN shows uneven row estimates per AMP
  • CPU utilization shows hot AMPs in Viewpoint
  • Query runs fine on small data, slow on full data

Resolution Strategies

  • Change PI to higher-cardinality column
  • Use composite PI — add 2nd column to PI
  • Add PARTITION BY to distribute within AMP
  • Use a hash banding column — artificial distribute column
🎯
Interview answer for "How do you handle skew?": "First I confirm skew using the HASHAMP(HASHROW()) query or DBC.TABLESIZE. If NUPI choice is the root cause, I evaluate changing PI to a higher-cardinality column or using a composite PI. For tables that can't change PI, I consider NUSI or query-level workarounds like explicit redistribution hints."
Query Tuning Best Practices

✅ DO These

  • Filter early — push WHERE conditions as deep as possible
  • Join on PI columns when possible
  • Use Volatile Tables for large intermediate results
  • Collect stats on all join/filter columns
  • Use PPI for large date-ranged tables
  • COLLECT STATS after large data loads
  • Use SET tables to prevent duplicates

❌ AVOID These

  • Functions on PI columns in WHERE: CAST(id AS CHAR)
  • Implicit type conversions on join columns
  • SELECT * on large tables
  • Product Joins (missing join conditions)
  • Low-cardinality PI (gender, status_cd, Y/N columns)
  • Running without stats (optimizer guesses wrong)
  • Subqueries when join will do

BTEQ — Batch Teradata Query

What is BTEQ?

BTEQ (Batch Teradata Query) is Teradata's primary command-line interface for submitting SQL, running ETL scripts, importing/exporting data, and scripting automation.

In production, nearly all scheduled Teradata jobs run through BTEQ scripts called from Unix shell scripts.

BTEQ Capabilities

  • Execute any Teradata SQL
  • Conditional logic (.IF / .GOTO)
  • Error handling via ERRORCODE
  • Data export to flat files
  • Data import from flat files
  • Format output (HTML, CSV, etc.)
  • Run from Unix/Windows shell

FastLoad

FastLoad Overview

  • Bulk load utility for empty tables only
  • Bypasses transient journal — no row-by-row logging
  • Best throughput for initial loads or truncate+reload
  • Cannot load to tables with secondary indexes (drop first)
  • Uses 2 phases: acquisition phase + application phase
  • Creates 2 error tables (ET and UV tables)

FastLoad Restrictions

  • ❌ Table must be EMPTY before loading
  • ❌ No secondary indexes on target table
  • ❌ No JOIN indexes allowed
  • ❌ No triggers on target table
  • ❌ No referential integrity constraints
  • ✅ Drop indexes before → recreate after load
FastLoadFastLoad script structure
/* FastLoad Script: load_sales.fl */
LOGON tdprod01/etl_user,passwd;

/* Define error tables */
BEGIN LOADING EDW_STG.SALES_FL_STG
    ERRORFILES EDW_STG.SALES_FL_ET, EDW_STG.SALES_FL_UV
    CHECKPOINT 50000;           /* restart checkpoint every 50k rows */

/* Define input file layout */
DEFINE
    SALE_ID     (INTEGER),
    CUST_ID     (INTEGER),
    SALE_DATE   (CHAR(10)),
    AMOUNT      (FLOAT),
    REGION_CD   (CHAR(3))
FILE=/data/incoming/sales_bulk.dat;

/* Insert statement */
INSERT INTO EDW_STG.SALES_FL_STG (
    SALE_ID, CUST_ID,
    CAST(:SALE_DATE AS DATE FORMAT 'YYYY-MM-DD'),
    :AMOUNT, :REGION_CD
);

END LOADING;
LOGOFF;
MultiLoad

MultiLoad Overview

  • Loads data into non-empty tables
  • Supports INSERT, UPDATE, DELETE, UPSERT
  • Can process multiple tables in one run
  • Best for incremental delta loads
  • Uses work tables and error tables
  • Slower than FastLoad but more flexible
  • 5 phases: Preliminary → DML Transaction → Acquisition → Application → Cleanup

FastLoad vs MultiLoad vs TPT

  • FastLoad: Empty tables, highest speed
  • MultiLoad: Non-empty, DML operations, delta loads
  • TPT: Modern successor, combines all — use for new scripts
  • BTEQ IMPORT: Small files, full flexibility
TPT — Teradata Parallel Transporter
🚄
TPT is the modern replacement for FastLoad, MultiLoad, and BTEQ exports. It uses a streaming operator model and is the recommended utility for new development.

TPT Operators

  • Load Operator — Replaces FastLoad
  • Update Operator — Replaces MultiLoad
  • Export Operator — High-speed export
  • Stream Operator — Real-time/micro-batch inserts
  • DataConnector — File I/O operator
  • DDL Operator — Create/drop tables

TPT Advantages

  • Single unified scripting language (TBUILD)
  • Parallel execution of operators
  • Supports cloud (S3, Azure Blob) directly
  • Better error handling and restart
  • Used in Azure Data Factory Teradata connector

DEFINE JOB LOAD_SALES_JOB
DESCRIPTION 'Load sales data to Teradata'
(
    DEFINE SCHEMA SALES_SCHEMA
    (
        SALE_ID   INTEGER,

        VARCHAR FileName = '/data/sales.csv',
        VARCHAR TextDelimiter = ',',
        VARCHAR Format = 'Delimited',
        VARCHAR OpenMode = 'Read'
    );

    DEFINE OPERATOR TD_LOADER
    TYPE LOAD
    SCHEMA SALES_SCHEMA
    ATTRIBUTES
    (
        VARCHAR TdpId       = 'tdprod01',
        VARCHAR UserName    = 'etl_user',

            :SALE_ID, :CUST_ID, :SALE_DATE, :AMOUNT, :REGION);')

Slowly Changing Dimensions (SCD)
🎯
Most asked advanced interview topic! Know SCD Type 1, 2, and 3 cold — especially the SQL to implement Type 2.

SCD Type 1 — Overwrite

Simply UPDATE the existing record. No history kept. Use when history doesn't matter.

SimplestNo History

SCD Type 2 — Full History

Add a NEW row with new values. Mark old row as expired. Full change history preserved.

Most CommonComplex SQL

SCD Type 3 — Limited History

Add columns: CURRENT_VAL and PREVIOUS_VAL. Only keeps 1 prior value.

Limited UseSimple
Teradata SQLSCD Type 2 Implementation
-- SCD Type 2: CUST_DIM structure
CREATE SET TABLE EDW_PROD.CUST_DIM
(
    CUST_SK       INTEGER NOT NULL,    -- surrogate key
    CUST_ID       INTEGER NOT NULL,    -- natural/business key
    CUST_NAME     VARCHAR(100),
    CITY          VARCHAR(50),
    SEGMENT       CHAR(10),
    EFF_FROM_DT   DATE NOT NULL,       -- when this version became active
    EFF_TO_DT     DATE,                -- NULL = current record
    IS_CURRENT    CHAR(1) DEFAULT 'Y'  -- Y = active, N = expired
) PRIMARY INDEX (CUST_ID);

-- Step 1: Expire changed records
UPDATE tgt
FROM EDW_PROD.CUST_DIM tgt,
     EDW_STG.CUST_STG  src
SET
    tgt.EFF_TO_DT  = DATE - 1,
    tgt.IS_CURRENT = 'N'
WHERE
    tgt.CUST_ID    = src.CUST_ID
    AND tgt.IS_CURRENT = 'Y'
    AND (tgt.CITY <> src.CITY OR tgt.SEGMENT <> src.SEGMENT);  -- changed

-- Step 2: Insert new version for changed + new customers
INSERT INTO EDW_PROD.CUST_DIM
    (CUST_SK, CUST_ID, CUST_NAME, CITY, SEGMENT, EFF_FROM_DT, EFF_TO_DT, IS_CURRENT)
SELECT
    NEXT_SK(),                  -- generate new surrogate key
    src.CUST_ID,
    src.CUST_NAME,
    src.CITY,
    src.SEGMENT,
    DATE,                       -- effective from today
    NULL,                       -- still current, no end date
    'Y'
FROM EDW_STG.CUST_STG src
WHERE NOT EXISTS (             -- no current matching record = new or just expired
    SELECT 1 FROM EDW_PROD.CUST_DIM tgt
    WHERE tgt.CUST_ID = src.CUST_ID
      AND tgt.IS_CURRENT = 'Y'
);
Temporal Tables

What are Temporal Tables?

Teradata natively supports bitemporal tables — tables that track both valid time (when a fact was true in the real world) and transaction time (when it was recorded in the database). This eliminates complex SCD-2 code.

Valid Time

When something was true in the real world. "John lived in Mumbai from 2020 to 2022."

Defined with VALIDTIME period columns.

Transaction Time

When data was stored in the database. "This record was inserted on 2020-01-15 and was never corrected."

Defined with TRANSACTIONTIME system period.

Teradata SQLTemporal table creation and query
-- Create a ValidTime temporal table
CREATE TABLE EDW_PROD.CUST_TEMPORAL
(
    CUST_ID   INTEGER NOT NULL,
    CUST_NAME VARCHAR(100),
    CITY      VARCHAR(50),
    VT        PERIOD(DATE) NOT NULL AS VALIDTIME
) PRIMARY INDEX (CUST_ID);

-- Sequenced insert: valid for a period
SEQUENCED VALIDTIME PERIOD FOR VT('2024-01-01', '2025-01-01')
INSERT INTO EDW_PROD.CUST_TEMPORAL (CUST_ID, CUST_NAME, CITY)
VALUES (1001, 'John Smith', 'Mumbai');

-- Query as-of a specific date
VALIDTIME AS OF DATE '2024-06-15'
SELECT * FROM EDW_PROD.CUST_TEMPORAL
WHERE CUST_ID = 1001;

-- Temporal UPDATE: automatically expires old, inserts new version
SEQUENCED VALIDTIME
UPDATE EDW_PROD.CUST_TEMPORAL
SET CITY = 'Bengaluru'
WHERE CUST_ID = 1001;
